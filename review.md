评审范围（按优先级从高到低）
================================

------------------------------------------------
1. 功能正确性（最高优先级）
------------------------------------------------

只查会真实导致错误的问题，且**必须给出 文件:行号 与 触发场景**：

- 逻辑错误 / 边界条件遗漏 / 状态机错误

------------------------------------------------
2. AI 过度设计专项（重点）
------------------------------------------------

对本库逐处判断下列"过度"是否真实存在；若存在，说明**删掉后是否影响功能**：

① 过度封装（一行 HAL 拆三层函数）
② 过度抽象（只有一种实现，却设计一整套接口）
③ 过度配置（增加永不关闭的 ENABLE_/USE_/SUPPORT_ 宏）
④ 过度泛化（按 255 个设备设计，实际只有一个）
⑤ 过度防御（每个函数都 if(NULL)/if(init)/if(index>=MAX)，但条件永不触发）
⑥ 过度模块化（拆出只被调用一次的小函数）
⑦ 过度状态机（简单逻辑被写成复杂状态机）
⑧ 过度注释（注释仅复读代码本身）
⑨ 过度兼容（为不存在的平台增加 #ifdef）
⑩ 过度可扩展（为不存在的 DMA/SPI/其他 RTOS 预留）

------------------------------------------------
3. 临界区专项
------------------------------------------------

逐处检查所有 taskENTER_CRITICAL / taskEXIT_CRITICAL / portENTER_CRITICAL / portEXIT_CRITICAL：

① 是否真的需要
② 保护范围是否过大（应只保护单个变量）
③ 能否改为 atomic 操作 / volatile
④ 是否影响实时性
⑤ 临界区内是否调用了 HAL / printf / memcpy / 复杂计算 / 循环 / 等待（这些都应指出）

最后评价：整个库的临界区设计是否合理。

------------------------------------------------
4. 运行时专项（FreeRTOS + STM32 + 实时性）
------------------------------------------------

聚焦真实隐患，不列空清单：

- HAL 使用是否正确（GPIO / TIM / NVIC / 中断）
- 任务 / Timer / Queue / ISR 使用是否符合 FreeRTOS 最佳实践
- 是否存在：关中断过长、阻塞、忙等待、长循环、优先级反转、CPU 浪费

------------------------------------------------
5. API 与架构
------------------------------------------------

- API 是否易理解、命名是否统一

- 判断标准同核心禁令：简单稳定 > 理论完美

------------------------------------------------
6. 可维护性与开源成熟度
------------------------------------------------

- 是否易读、变量命名是否统一、函数长度 / 文件划分是否合理
